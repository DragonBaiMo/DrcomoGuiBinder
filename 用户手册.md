# DrcomoGuiBinder — 最终用户使用手册（初稿）

---

## 1. 功能简介

DrcomoGuiBinder 是一个用于在游戏内通过 GUI 为“主界面（Main）槽位”绑定“子界面（Sub）条目值”的插件模块。核心作用如下：

* 玩家通过点击 Main 的槽位进入 Sub，选择 Sub 条目后将该条目的字符串值绑定到该玩家的 Main 指定槽位；
* 绑定后 Main 槽的显示完全由 Sub 提供的展示（display）控制，支持针对不同 Main 的展示覆盖；
* 提供占位符供其他系统或消息使用，格式简单易用，支持默认值与二次解析；
* 支持玩家级别独立绑定并持久化保存，提供管理/清理/命令操作入口。

典型适用场景：装备/技能槽位个性化配置、玩家自定义变量选择、插件间占位符值配置等。

---

## 2. 功能入口说明

用户可通过以下方式访问与使用本模块：

1. **游戏内命令（主要入口）**

    * 打开 Main 界面：`/DrcomoGuiBinder open <mainId>`
      例：`/DrcomoGuiBinder open weapons`
    * 管理与调试命令（仅具权限者使用）：详见 速查表。

2. **在游戏内 GUI 操作**

    * 在打开的 Main 界面点击某个槽位（Slot）会打开该槽关联的 Sub 界面。
    * 在 Sub 界面点击某条目（Entry）会将该条目的值绑定回 Main 的原槽，并返回 Main 并只刷新该槽的显示。

3. **文件配置（供管理员配置）**

    * Main 文件位置：`plugins/<plugin>/Main/<mainId>.yml`
    * Sub 文件位置：`plugins/<plugin>/Sub/<subId>.yml`
    * 配置为每个 GUI 单独文件，管理员通过编辑这些文件来定义界面与条目。

注：普通玩家无需编辑配置文件即可使用。管理员用于初始化界面与条目配置。

## 3. 操作流程详解（面向最终用户）

下面以完整步骤描述玩家从打开 Main 到绑定 Sub 的典型流程和可选操作。

### 3.1 打开 Main 界面

步骤：

1. 在聊天框输入命令：`/DrcomoGuiBinder open <mainId>`（例如 `weapons`）。
2. 系统弹出 Main 界面，界面上显示若干格（槽位），每个槽有提示文本或空槽样式。

说明：

* 若您不知道可用的 `<mainId>`，联系管理员或使用 `/DrcomoGuiBinder listmains`（管理员命令）查询。
* 部分服务器可在特定 NPC/方块/菜单中绑定打开该命令，按服务器提示操作。

### 3.2 从 Main 打开 Sub（选择要绑定的槽）

步骤：

1. 在 Main 界面中，将鼠标移到想要绑定的槽上。
2. 单击该槽（通常是左键点击）。
3. 系统会打开与该槽关联的 Sub 界面。

说明：

* 如果该槽已绑定，会显示当前绑定结果（由 Sub 决定展示）。若想修改绑定，直接点击该槽并从 Sub 选择新条目即可覆盖原绑定。
* 如果点击后无反应，可能是该槽未配置 Sub 或权限受限，请联系管理员。

### 3.3 在 Sub 中选择条目（完成绑定）

步骤：

1. 在 Sub 界面中找到想要的条目（每个条目有图标、名称和说明）。
2. 单击该条目。
3. 系统会将该条目的 `key` 和 `value` 记录为玩家在该 Main 的该槽的绑定值。
4. 系统自动返回 Main 界面并**仅刷新该槽的显示**，显示基于 Sub 的 `display`（或按 Main 特定覆盖规则显示）。

说明：

* 绑定前可以查看条目的 `display` 和 `value` 说明。
* 部分 Sub 条目的 value 支持占位符（如 `%player_name%`），绑定后最终显示或占位符解析可能在渲染时完成（视管理员配置）。

### 3.4 清除或手动修改绑定

常用方式：

* 在游戏内使用命令清除：`/DrcomoGuiBinder clear <mainId> <slot>`。例：`/DrcomoGuiBinder clear weapons 10`。
* 也可重新绑定：重复上述步骤，从 Sub 选择另一个条目以覆盖。

权限：

* 清除他人绑定或批量操作通常需要管理员权限。

## 4. 配置与个性化（为管理员准备，但同时以“用户可理解”的方式说明）

> 下面给出**完整独立的配置文件示例**，管理员直接按文件替换并重载即可。普通玩家不必编辑配置，但应了解展示行为与占位符含义。

### 4.1 Main 配置字段说明（完整文件示例）

**文件路径示例**：`plugins/DrcomoGuiBinder/Main/weapons.yml`
**完整文件（独立可用）**：

```yaml
id: weapons
title: "&8武器主界面"
size: 54

# 每个槽声明其对应的 Sub id，和未绑定时的显示(displayEmpty)
slots:
  - slot: 10
    id: "weapon_slot_1"
    sub: sword_vars
    displayEmpty:
      material: STONE
      name: "&7未绑定"
      lore:
        - "&f点击选择"

  - slot: 11
    id: "weapon_slot_2"
    sub: bow_vars
    displayEmpty:
      material: BOW
      name: "&7未绑定"
```

**字段说明（面向管理员但简明）**：

* `id`：Main 的唯一标识（供命令使用，如 `weapons`）。
* `title`：界面顶部标题（支持颜色代码）。
* `size`：界面格子总数（通常为 9 的倍数）。
* `slots`：列表，每项为一个槽配置：

    * `slot`：槽位索引（0 起或管理员约定，示例采用 0/1/.. 或 1/..，以实际服务器为准）。
    * `id`（可选）：槽位的字符串别名，供占位符或脚本引用；填写后可使用 `%dgb_*` 占位符时用该别名代替数字槽位。
    * `sub`：该槽点击后将打开的 Sub 的 id（Sub 文件名中的 `id` 字段）。
    * `displayEmpty`：槽位在未绑定时的展示，包含 `material`、`name`、`lore` 等字段。

### 4.2 Sub 配置字段说明（完整文件示例）

**文件路径示例**：`plugins/DrcomoGuiBinder/Sub/sword_vars.yml`
**完整文件（独立可用）**：

```yaml
id: sword_vars
title: "&8选择剑类变量"
size: 54

entries:
  - slot: 20
    key: "damage"
    value: "12"
    display:
      material: IRON_SWORD
      name: "&f伤害={value}"
      lore:
        - "&7数值绑定"
        - "&7键: {key}"

    # 当被不同 Main 使用时，可以指定覆盖 display
    overrides:
      weapons:
        display:
          name: "&a武器伤害={value}"
          glow: true

  - slot: 21
    key: "damage"
    value: "16"
    display:
      material: DIAMOND_SWORD
      name: "&f伤害={value}"
```

**字段说明**：

* `id`：Sub 的唯一标识（例如 `sword_vars`）。
* `title`、`size`：界面标题与大小。
* `entries`：条目列表，每条包含：

    * `slot`：Sub 中显示的位置索引。
    * `key`：条目的键（用于语义化识别，可重复）。
    * `value`：要绑定的字符串值。可以是文本或包含占位符的字符串（如 `%player_name%`）。
    * `display`：Sub 条目在绑定后用于渲染 Main 槽的展示字段（material/name/lore/customModelData/flags/glow 等）。
    * `overrides`（可选）：当该 entry 被某个 Main 绑定时，针对特定 `mainId` 的覆盖 `display`。格式为 `overrides: <mainId>: display: {...}`。

### 4.3 全局配置（示例）

**文件路径**：`plugins/DrcomoGuiBinder/config.yml`（示例独立文件）

```yaml
# DrcomoGuiBinder 主配置
logLevel: INFO
resolveAtBind: false
clickCooldownMs: 300
sessionTimeoutMs: 90000

# 数据库配置
database:
  # 数据库类型：sqlite 或 mysql
  type: sqlite
  
  # SQLite 配置（当 type=sqlite 时）
  file: data.db
  
  # MySQL 配置（当 type=mysql 时）
  # host: localhost
  # port: 3306
  # database: drcomoguibinder
  # username: root
  # password: ""

async:
  writerThreads: 2

placeholders:
  prefix: dgb
```

说明：

* `resolveAtBind=false` 推荐用于 value 中包含动态占位符（例如 `%player_name%`），使最终展示与玩家状态保持一致；设置为 `true` 会在绑定时把当时解析结果写入数据库并以后固定。
* **数据库配置**：
  - `type`: 选择数据库类型，支持 `sqlite`（默认）和 `mysql`
  - **SQLite 配置**：仅需 `file` 指定数据库文件名
  - **MySQL 配置**：需要完整的连接信息，包括 `host`、`port`、`database`、`username`、`password`

### 4.4 语言文件（提示文本）示例（独立）

**文件路径**：`plugins/DrcomoGuiBinder/lang.yml`

```yaml
bind-success: "&a绑定成功: {main}[{slot}] ← {sub}.{key}={value}"
bind-cleared: "&c已清除 {main}[{slot}] 的绑定"
open-no-exist: "&c错误: 主界面 {main} 不存在"
sub-no-exist: "&c错误: 子界面 {sub} 不存在"
permission-denied: "&c你没有权限执行此操作"
```

管理员可自定义这些文案。

## 5. 错误提示与异常处理（面向最终用户）

本节列举玩家可能遇到的常见情形、插件提示及用户可执行操作。

### 5.1 常见提示与解决办法

1. **提示**：`错误: 主界面 <mainId> 不存在`

    * 含义：您尝试打开的 Main ID 未被配置或文件缺失。
    * 用户操作：联系管理员确认 `<mainId>` 是否正确或是否已加载配置。管理员可使用 `/DrcomoGuiBinder reload` 重载配置。

2. **提示**：`错误: 子界面 <subId> 不存在`（点击 Main 某槽后发生）

    * 含义：该 Main 槽所关联的 Sub 配置缺失。
    * 用户操作：联系管理员修复 Sub 文件或更改该槽关联。

3. **提示**：`你没有权限执行此操作`（在尝试清除他人绑定或使用管理命令时）

    * 含义：您尝试的命令需要更高权限。
    * 用户操作：使用自己的槽位与绑定；清理或跨玩家操作请联系管理员。

4. **情形**：点击槽位无反应

    * 可能原因：槽未配置 Sub、会话超时、网络延迟或插件出错。
    * 用户操作：稍等后重试；若持续无效，记录时间与命令交互并反馈管理员。

5. **情形**：绑定后 Main 显示不正确或未更新

    * 可能原因：Sub 的 `display` 配置项缺失或格式错误；管理员启用了 `resolveAtBind` 并已固化旧值。
    * 用户操作：退出并重新打开 Main 界面尝试刷新；若仍不正常，截图并反馈管理员检查 Sub 配置。

### 5.2 管理员常见告警与处理建议（仅简要）

* 数据库写入失败：检查磁盘空间与文件权限。
* 配置热重载后显示异常：检查 YAML 语法（缩进）或 `id` 书写一致性。

## 6. 使用建议与最佳实践（面向最终用户与管理员）

### 给普通玩家的建议

* 在绑定前确认 Sub 条目的说明（lore），避免绑定错误值。
* 若值包含占位符（例如 `%player_name%`），确认服务器配置的解析策略（绑定时固化或渲染时解析）。
* 想要更换绑定值时直接在 Main 点击对应槽并选择新条目，操作简单直接。

### 给管理员的建议

* 命名规范：`mainId` 与 `subId` 尽量使用短小英文或拼音，便于命令使用与文案替换（例如 `weapons`, `sword_vars`）。
* Sub 优先提供 `display` 字段，Main 仅负责 `displayEmpty`，减少配置重复。
* 使用 `overrides` 提供针对特定 Main 的显示差异，不要在 Main 中重复定义绑定态样式。
* 对于动态值优先设 `resolveAtBind: false`，保持实时解析，避免数据过期。
* 定期备份 `plugins/<plugin>/` 目录下的 `data.db` 与 `Main`/`Sub` 配置文件。

## 7. 快速参考表（命令与占位符速查）

### 7.1 常用命令（管理员/玩家）

```
/DrcomoGuiBinder open <mainId> [player]      # 打开 Main 界面；player 参数为管理员打开他人界面
/DrcomoGuiBinder clear <mainId> <slot> [player]  # 清除绑定（可指定玩家）
/DrcomoGuiBinder bind <mainId> <slot> <subId> <entryKey>  # 手动绑定（调试用）
/DrcomoGuiBinder list <player>               # 列出玩家所有绑定（管理员可用）
/DrcomoGuiBinder reload                      # 重载配置与语言
/DrcomoGuiBinder save                        # 强制持久化缓存到数据库
```

### 7.2 占位符速查（在文本中使用）

* `%dgb_value_<mainId>_<slot或槽位ID>%` → 返回绑定的 `value`（若未绑定则空或默认）。
* `%dgb_key_<mainId>_<slot或槽位ID>%` → 返回绑定的 `key`。
* `%dgb_id_<mainId>_<槽位ID>%` → 以槽位的字符串 `id` 访问绑定的 `key`，语义等同于 `key` 功能，便于按别名写法（例：`%dgb_id_class_class1%` → `哈哈`）。
* `%dgb_display_<mainId>_<slot或槽位ID>%` → 返回绑定后用于展示的名称（解析后）。
* `%dgb_sub_<mainId>_<slot或槽位ID>%` → 返回该槽当前绑定的 Sub `id`，未绑定时返回空或默认值。
* 所有占位符均支持默认值格式：`%dgb_value_<mainId>_<slot或槽位ID>|默认文本%`。
* 若 Main 槽配置了 `id` 字段，可在占位符中以该别名替代数字槽位（例如 `%dgb_sub_class_class1%`）。

### 7.3 文件路径

```
plugins/DrcomoGuiBinder/Main/<mainId>.yml
plugins/DrcomoGuiBinder/Sub/<subId>.yml
plugins/DrcomoGuiBinder/config.yml
plugins/DrcomoGuiBinder/lang.yml
plugins/DrcomoGuiBinder/data.db      # SQLite 数据库文件（默认）
plugins/DrcomoGuiBinder/schema.sql   # SQLite 表结构脚本
plugins/DrcomoGuiBinder/schema-mysql.sql  # MySQL 表结构脚本
```

## 8. FAQ（常见问题）

Q1：绑定后其他玩家能看到我的绑定吗？
A1：Main 槽的绑定为玩家级别独立。其他玩家不会看到你的私有绑定，除非管理员查看或导出数据。

Q2：绑定的是即时值还是模板？占位符会变化吗？
A2：取决于管理员在 `config.yml` 的 `resolveAtBind` 设置：

* `true`：绑定时解析并固化当前值；以后不再变化。
* `false`（推荐）：渲染时再解析占位符，保持动态更新。

Q3：如何把 Sub 的条目显示作用于不同 Main 而不同样式显示？
A3：在 Sub 的条目中使用 `overrides` 字段，按 `mainId` 定义对应的 `display`，系统会优先使用 overrides 中的 display 覆盖默认 display。

Q4：我误删/误绑定了，如何恢复？
A4：可在游戏中用 `/DrcomoGuiBinder clear <mainId> <slot>` 清除该槽绑定；若误操作影响较广，请联系管理员恢复备份。

Q5：我想批量替换某类绑定值，有快捷方式吗？
A5：插件提供基本命令，但不提供复杂批量 UI。建议管理员导出数据后用脚本处理再导入（需管理员权限和技术支持）。

Q6：为什么我点击槽没有打开 Sub？
A6：可能原因：该槽未配置 Sub、该 Main 未加载、或权限受限。请截图并联系管理员检查对应的 Main/Sub 文件。

Q7：如何从 SQLite 切换到 MySQL？
A7：管理员需要：
1. 在 config.yml 中将 `database.type` 改为 `mysql`
2. 配置 MySQL 连接信息（host、port、database、username、password）
3. 在 MySQL 中创建对应数据库
4. 重启服务器（插件会自动创建表结构）
5. 如需迁移现有数据，可联系技术人员协助

Q8：MySQL 连接失败怎么办？
A8：检查以下项目：
- MySQL 服务是否正常运行
- 数据库名称、用户名、密码是否正确
- 网络连通性（特别是远程 MySQL）
- MySQL 用户是否有足够权限（需要 CREATE、SELECT、INSERT、UPDATE、DELETE 权限）
- 查看服务器日志获取具体错误信息
