## 1. 核心概念与工作流程

### 1.1. 功能概述

`DrcomoGuiBinder` 是一款高度可配置的 Minecraft GUI 绑定插件。它的核心功能是让玩家通过图形界面（GUI）将一个“选项”与一个“槽位”进行绑定，并将这个绑定关系以 PlaceholderAPI (PAPI) 占位符的形式暴露给其他插件使用。

这个机制允许服务器管理员创建复杂的、可由玩家自定义的交互式面板，而无需编写任何代码。例如，玩家可以通过 GUI 选择自己的职业、称号、性别、阵营等，这些选择的结果会动态地反映在聊天、计分板或其他依赖 PAPI 的系统中。

### 1.2. 核心三要素

插件的所有功能都围绕以下三个核心配置文件展开：

1.  **主界面 (Main GUI)**

      * **定义**：一个容器界面，定义了多个可供绑定的“**槽位 (Slot)**”。
      * **作用**：作为用户交互的入口。每个槽位都像一个“插座”，等待用户插入一个“电器”。
      * **示例**：一个“角色面板”界面，其中有“职业”、“称号”、“阵营”等不同的槽位。

2.  **子界面 (Sub GUI)**

      * **定义**：一个选项列表界面，包含了多个可供选择的“**条目 (Entry)**”。
      * **作用**：当玩家点击主界面的某个槽位时，会弹出其关联的子界面，展示所有可用选项。
      * **示例**：点击“职业”槽位后，弹出一个包含“战士”、“法师”、“弓箭手”等选项的列表。

3.  **绑定 (Binding)**

      * **定义**：玩家在子界面中选择一个条目后，该条目的 `key` 和 `value` 就会与最初点击的主界面槽位建立一对一的关联关系。
      * **作用**：这个关系会被持久化存储在数据库中，并可以通过插件提供的 PAPI 占位符随时调用。
      * **示例**：玩家在“职业”子界面选择了“战士”条目，系统就记录下 `玩家A` 的 `角色面板` 的 `职业槽位` 绑定了 `战士` 这个值。

### 1.3. 完整工作流程

对于使用者（玩家/管理员）而言，一个完整的交互流程如下：

1.  **管理员配置**：管理员创建 Main 和 Sub 的 YAML 配置文件，定义好界面布局、槽位、可选条目以及它们之间的关联。
2.  **玩家打开主界面**：玩家通过指令 `/dgb open <mainId>` 打开一个主界面。
3.  **玩家选择槽位**：玩家点击主界面上某个感兴趣的槽位（例如“称号”槽）。
4.  **系统打开子界面**：插件根据该槽位的配置，打开其关联的子界面（例如 `titles.yml`）。
5.  **玩家选择条目**：玩家在子界面中点击一个条目（例如“英勇的战士”）。
6.  **系统创建绑定**：
      * 插件捕获该点击事件。
      * 将该条目的 `key` 和 `value` 记录下来。
      * 在数据库中为该玩家、该主界面、该槽位创建一个绑定记录。
7.  **系统刷新界面**：
      * 自动关闭子界面，返回到主界面。
      * 主界面上刚刚被点击的槽位，其显示内容会根据所选条目的 `display` 配置进行刷新。
8.  **外部系统调用**：
      * 其他插件（如聊天插件）通过 PAPI 请求 `%dgb_value_角色面板_称号%` 这样的占位符。
      * `DrcomoGuiBinder` 插件监听到请求，查询数据库，返回该玩家绑定的值（例如 `&6[英勇的战士]`），最终显示在聊天栏中。

## 2. 全局配置 (`config.yml`)

这是插件的核心配置文件，用于控制插件的全局行为、数据库连接和性能参数。

### 2.1. 配置结构说明

| 键 (Key) | 类型 | 描述 | 默认值 |
| :--- | :--- | :--- | :--- |
| `logLevel` | 字符串 | 控制插件在控制台输出的日志详细程度。可选值为 `INFO`, `DEBUG`, `WARN`, `ERROR`。 | `INFO` |
| `resolveAtBind` | 布尔值 | 控制占位符解析时机。`true`: 在玩家点击绑定时立即解析 `value` 中的PAPI占位符并存入数据库（静态值）；`false`: 存储原始 `value` 字符串，每次通过PAPI请求时再实时解析（动态值）。 | `false` |
| `clickCooldownMs` | 整数 | 玩家在GUI中两次点击之间的最小间隔时间（毫秒），用于防止过快点击导致意外行为。 | `300` |
| `sessionTimeoutMs` | 长整数 | GUI会话的超时时间（毫秒）。玩家打开GUI后若长时间无操作，会话将失效。 | `90000` |
| `database.type` | 字符串 | 选择用于存储绑定数据的数据库类型。支持 `sqlite` 和 `mysql`。 | `sqlite` |
| `database.file` | 字符串 | 当 `type` 为 `sqlite` 时，指定数据库文件的路径（相对于插件文件夹）。 | `data.db` |
| `database.host` | 字符串 | 当 `type` 为 `mysql` 时，指定MySQL服务器地址。 | `localhost` |
| `database.port` | 整数 | 当 `type` 为 `mysql` 时，指定MySQL服务器端口。 | `3306` |
| `database.database` | 字符串 | 当 `type` 为 `mysql` 时，指定要使用的数据库名称。 | `drcomoguibinder` |
| `database.username` | 字符串 | 当 `type` 为 `mysql` 时，指定数据库连接用户名。 | `root` |
| `database.password` | 字符串 | 当 `type` 为 `mysql` 时，指定数据库连接密码。 | `""` |
| `async.writerThreads` | 整数 | 用于处理数据库异步写入操作的线程池大小。 | `2` |
| `placeholders.prefix` | 字符串 | 插件注册的所有 PAPI 占位符的前缀。 | `dgb` |

### 2.2. 示例配置文件

**SQLite (默认)**

```yaml
# DrcomoGuiBinder 主配置

# 日志级别: INFO, DEBUG, WARN, ERROR
logLevel: INFO

# 是否在绑定时解析 value 中的 PAPI 变量
# false: 每次通过占位符获取时实时解析 (推荐，动态)
# true: 绑定时解析一次并存入数据库 (静态)
resolveAtBind: false

# GUI 点击冷却时间 (毫秒)
clickCooldownMs: 300

# GUI 会话超时时间 (毫秒)
sessionTimeoutMs: 90000

# 数据库配置
database:
  # 数据库类型：sqlite 或 mysql
  type: sqlite

  # SQLite 配置（当 type=sqlite 时）
  file: data.db

# 异步写入数据库的线程数
async:
  writerThreads: 2

# PlaceholderAPI 占位符前缀
placeholders:
  prefix: dgb
```

**MySQL**

```yaml
# ... 其他配置 ...

# 数据库配置
database:
  # 数据库类型：sqlite 或 mysql
  type: mysql

  # MySQL 配置（当 type=mysql 时）
  host: "127.0.0.1"
  port: 3306
  database: "minecraft_server"
  username: "plugin_user"
  password: "your_secure_password"

# ... 其他配置 ...
```

## 3. 主界面配置 (Main GUI)

主界面是用户交互的起点，定义了可绑定的“槽位”。

### 3.1. 文件位置

所有主界面配置文件都应放置在 `plugins/DrcomoGuiBinder/Main/` 目录下。每个 `.yml` 文件代表一个独立的主界面。文件名（不含`.yml`后缀）可以作为其默认 `id`。

### 3.2. 配置结构说明

一个主界面配置文件主要由以下几个部分组成：

| 键 (Key) | 类型 | 是否必需 | 描述 |
| :--- | :--- | :--- | :--- |
| `id` | 字符串 | 否 | 主界面的唯一ID。用于 `/dgb open` 命令和占位符中。如果未提供，将使用文件名作为ID。 |
| `title` | 字符串 | 是 | GUI 顶部的标题，支持 Minecraft 颜色代码 (`&`)。 |
| `size` | 整数 | 是 | GUI 的大小（格子总数），必须是 9 的倍数（例如 9, 18, 27, 36, 45, 54）。 |
| `slots` | 列表 | 是 | 定义所有可交互槽位的列表。 |

#### `slots` 列表项结构

`slots` 列表中的每一个元素都是一个对象，用于定义一个具体的槽位。

| 键 (Key) | 类型 | 是否必需 | 描述 |
| :--- | :--- | :--- | :--- |
| `slot` | 整数 | 是 | 槽位的索引，从 0 开始计数（第一行是 0-8，第二行是 9-17，以此类推）。 |
| `id` | 字符串 | 否 | 槽位的唯一字符串ID（别名）。设置后，可以在 PAPI 占位符中使用这个ID代替数字索引，使占位符更具可读性。 |
| `type` | 字符串 | 否 | 槽位类型，可选值：`DEFAULT`（默认，正常绑定槽）、`DECORATION`（装饰槽，仅显示不可点击）。装饰槽应省略 `sub` 字段。 |
| `sub` | 字符串 | 条件 | 点击此槽位后要打开的子界面ID（对应 `Sub/` 目录下的某个配置文件的 `id`）。**当 `type` 为 `DECORATION` 时不需要**。 |
| `displayEmpty` | 对象 | 是 | 定义当此槽位**未被任何条目绑定时**所显示的物品样式。这是一个标准的 [**物品模板 (`ItemTemplate`)**](https://www.google.com/search?q=%2333-%E7%89%A9%E5%93%81%E6%A8%A1%E6%9D%BF-itemtemplate-%E8%AF%A6%E8%A7%A3)。若未配置，装饰槽默认为灰色玻璃板。 |

### 3.3. 物品模板 (`ItemTemplate`) 详解

`ItemTemplate` 是一个标准化的对象结构，用于定义 Minecraft 中的物品外观。它被用于 `displayEmpty`、Sub GUI 的 `display` 和 `overrides` 中。

| 键 (Key) | 类型 | 描述 | 示例 |
| :--- | :--- | :--- | :--- |
| `material` | 字符串 | 物品的材质。必须是 Spigot Material 枚举中的有效名称。 | `DIAMOND_SWORD`, `PLAYER_HEAD` |
| `amount` | 整数 | 物品的数量。默认为 1。 | `1` |
| `name` | 字符串 | 物品的显示名称。支持颜色代码。 | `&6勇者之剑` |
| `lore` | 字符串列表 | 物品的描述文本（lore），列表中的每个字符串代表一行。 | `- "&7攻击力 +10"`<br>`- "&f点击选择"` |
| `customModelData` | 整数 | 自定义模型数据 (CustomModelData)，用于资源包。 | `10001` |
| `flags` | 字符串列表 | 添加物品标志 (ItemFlag)，用于隐藏特定信息，如附魔、属性等。 | `- HIDE_ENCHANTS`<br>`- HIDE_ATTRIBUTES` |
| `glow` | 布尔值 | 是否让物品发光（附魔效果）。`true` 为发光。 | `true` |
| `nbt` | 对象 | (高级) 要写入物品的自定义 NBT 标签。键值对形式。 | `nbt:`<br>`   myplugin_weapontype: "sword" ` |
| `skullTexture` | 字符串 | (已弃用) 如果 `material` 是 `PLAYER_HEAD`，此项用于指定头颅的纹理值。当前版本不实现纹理渲染，会显示普通玩家头颅。 | |

### 3.4. 实际示例

此示例定义了一个名为 `character_panel` 的主界面，包含两个可绑定槽位：一个用于职业，一个用于称号。

**文件路径**: `plugins/DrcomoGuiBinder/Main/character_panel.yml`

```yaml
# 主界面的唯一ID，用于命令 /dgb open character_panel
id: character_panel
# GUI 顶部的标题
title: '&1角色面板'
# GUI 大小，3行27格
size: 27

# 定义界面中的所有可交互槽位
slots:
  - # 职业槽位
    slot: 11 # 放置在第12格 (索引从0开始)
    id: "class_slot" # 为这个槽位设置一个字符串ID，方便PAPI调用
    sub: "class_selection" # 点击后打开名为 class_selection 的子界面
    displayEmpty: # 当未绑定任何职业时，显示以下物品
      material: BOOK
      name: '&7[未选择职业]'
      lore:
        - '&f点击此处选择你的职业'
        - '&e职业决定了你的基础属性。'

  - # 称号槽位
    slot: 15 # 放置在第16格
    id: "title_slot" # 称号槽位的字符串ID
    sub: "title_selection" # 点击后打开名为 title_selection 的子界面
    displayEmpty: # 当未绑定任何称号时
      material: NAME_TAG
      name: '&7[未选择称号]'
      lore:
        - '&f点击此处选择一个称号'
```

## 4. 子界面配置 (Sub GUI)

子界面提供了具体的选项列表，供玩家选择并进行绑定。

### 4.1. 文件位置

所有子界面配置文件都应放置在 `plugins/DrcomoGuiBinder/Sub/` 目录下。每个 `.yml` 文件代表一个独立的子界面。

### 4.2. 配置结构说明

一个子界面配置文件由以下几个部分组成：

| 键 (Key) | 类型 | 是否必需 | 描述 |
| :--- | :--- | :--- | :--- |
| `id` | 字符串 | 否 | 子界面的唯一ID。如果未提供，将使用文件名作为ID。 |
| `title` | 字符串 | 是 | GUI 顶部的标题，支持颜色代码。 |
| `size` | 整数 | 是 | GUI 的大小，必须是 9 的倍数。 |
| `entries` | 列表 | 是 | 定义所有可选条目的列表。 |

#### `entries` 列表项结构

`entries` 列表中的每一个元素都是一个对象，定义了一个可供选择的条目。

| 键 (Key) | 类型 | 是否必需 | 描述 |
| :--- | :--- | :--- | :--- |
| `slot` | 整数 | 是 | 该条目在子界面中显示的位置索引（从0开始）。 |
| `type` | 字符串 | 否 | 条目类型，可选值：`DEFAULT`（默认，正常绑定条目）、`DECORATION`（装饰槽，仅显示不可点击）、`RETURN`（返回槽，点击后返回上一级主界面）。默认为 `DEFAULT`。 |
| `key` | 字符串 | 是 | 条目的唯一标识符（在一个 Sub GUI 内）。这个 `key` 会被存入数据库，并可以通过 `%dgb_key%` 占位符获取。**对于装饰槽和返回槽，此字段只用于配置内部标识，无实际作用**。 |
| `value` | 字符串 | 是 | 选择此条目后，实际绑定的值。这个 `value` 会被存入数据库，并可以通过 `%dgb_value%` 占位符获取。它支持 PAPI 占位符。**对于装饰槽和返回槽，可为空字符串**。 |
| `display` | 对象 | 是 | 该条目在**子界面中**的显示样式，同时也是绑定**成功后**在**主界面槽位**上显示的**默认样式**。这是一个标准的 [**物品模板 (`ItemTemplate`)**](https://www.google.com/search?q=%2333-%E7%89%A9%E5%93%81%E6%A8%A1%E6%9D%BF-itemtemplate-%E8%AF%A6%E8%A7%A3)。若未配置，装饰槽默认为灰色玻璃板，返回槽默认为箭头。 |
| `overrides` | 对象 | 否 | （高级功能）用于为特定的主界面覆盖 `display` 样式。详见 [**覆盖系统 (`overrides`)**](https://www.google.com/search?q=%2344-%E8%A6%86%E7%9B%96%E7%B3%BB%E7%BB%9F-overrides-%E8%AF%A6%E8%A7%A3)。**装饰槽和返回槽不支持该字段**。 |
| `conditions` | 对象 | 否 | （高级功能）用于设置条目的可见性和可选性条件。详见 [**条件系统 (`conditions`)**](https://www.google.com/search?q=%2343-%E6%9D%A1%E4%BB%B6%E7%B3%BB%E7%BB%9F-conditions-%E8%AF%A6%E8%A7%A3)。**装饰槽和返回槽不支持该字段**。 |

### 4.3. 条件系统 (`conditions`) 详解

通过 `conditions` 字段，可以为每个条目设置复杂的显示和选择条件。这些条件基于 PlaceholderAPI 表达式进行判断。

`conditions` 对象包含两个可选的子字段：

1.  **`display`**: (字符串列表)

      * **作用**：定义条目在子界面中**可见**所需满足的条件。
      * **逻辑**：列表中的所有条件都必须为 `true`（逻辑 `AND`），该条目才会被显示。如果列表为空或未定义，则默认可见。

2.  **`choose`**: (字符串列表)

      * **作用**：定义玩家**能够点击并选择**该条目所需满足的条件。
      * **逻辑**：列表中的所有条件都必须为 `true`（逻辑 `AND`），玩家才能成功绑定该条目。如果玩家点击了一个不满足选择条件的条目，会收到提示信息。

**条件表达式语法**：

  * **格式**：`"%placeholder% 操作符 '值'"` 或 `"%placeholder% 操作符 %another_placeholder%"`
  * **支持的操作符**：
      * 数值比较：`>`, `>=`, `<`, `<=`, `==`, `!=`
      * 字符串比较：`==` (等于), `!=` (不等于), `>>` (包含), `!>>` (不包含), `<<` (被包含), `!<<` (不被包含)
  * **逻辑组合**：支持使用 `&&` (与) 和 `||` (或) 来组合多个表达式，并可用 `()` 控制优先级。
  * **示例**：
      * `"%player_level% >= 20"` (玩家等级大于等于20)
      * `"%essentials_money_formatted% > 1000"` (玩家金钱大于1000)
      * `"%luckperms_primary_group_name% == 'vip'"` (玩家在 'vip' 权限组)
      * `"%player_world_name% >> 'resource' && %player_gamemode% != 'CREATIVE'"` (玩家在资源世界且不是创造模式)

### 4.4. 覆盖系统 (`overrides`) 详解

`overrides` 字段允许同一个子界面条目在被不同的主界面绑定时，呈现出不同的外观。这极大地增强了配置的复用性。

  * **结构**：`overrides` 是一个对象，其**键**是主界面的 `id`，**值**是一个包含 `display` 字段的对象。
  * **工作方式**：当一个条目被绑定到主界面的槽位时，插件会检查该条目的 `overrides` 字段。如果找到了与当前主界面 `id` 匹配的键，就会使用该键下定义的 `display` 模板来渲染主界面槽位的物品，而不是使用默认的 `display`。
  * **合并逻辑**：`override` 中的 `display` 会与默认的 `display` 进行合并。`override` 中定义的字段会覆盖默认 `display` 的同名字段，未定义的字段则保留默认 `display` 的设置。

### 4.5. 实际示例

此示例定义了一个名为 `class_selection` 的子界面，用于上面 `character_panel` 主界面的职业选择。

**文件路径**: `plugins/DrcomoGuiBinder/Sub/class_selection.yml`

```yaml
id: class_selection
title: '&2选择你的职业'
size: 36

entries:
  - # 战士条目
    slot: 10
    key: "warrior"
    value: '&c战士'
    display: # 默认显示样式
      material: IRON_SWORD
      name: '&c{value}' # {value} 会被替换为上面的 "value" 字段内容
      lore:
        - '&7近战物理输出职业。'
        - ''
        - '&a点击选择'
      flags:
        - HIDE_ATTRIBUTES
    overrides:
      character_panel: # 当被 id 为 'character_panel' 的主界面绑定时
        display: # 使用这个 display 覆盖默认的
          glow: true # 让物品发光
          name: '&c[职业] {value}' # 修改名称格式

  - # 法师条目
    slot: 11
    key: "mage"
    value: '&b法师'
    conditions: # 条件设置
      display: # 显示条件
        - "%player_has_permission_magic.use% == 'yes'" # 只有拥有 magic.use 权限的玩家才能看到这个选项
      choose: # 选择条件
        - "%player_level% >= 10" # 玩家等级必须大于等于10才能选择
    display:
      material: BOOK
      name: '&b{value}'
      lore:
        - '&7远程魔法输出职业。'
        - '&c需要等级达到 10 级才能选择。'
        - '&c需要拥有 "magic.use" 权限。'
        - ''
        - '&a点击选择'
```

## 6. 占位符 (Placeholders)

插件通过 PlaceholderAPI 提供了丰富的占位符，用于在其他插件中获取玩家的绑定数据。

### 6.1. 语法结构

所有占位符的基本组成均为：

`%<前缀>_<功能>_<参数1>_<参数2>[|默认值]%`

其中：

  * **`<前缀>`**: 在 `config.yml` 中定义的 `placeholders.prefix`，默认为 `dgb`。
  * **`<功能>`**: 决定要获取哪种数据，例如 `value`, `key` 等。
  * **`<参数1>` 与 `<参数2>`**: 根据功能不同，对应的含义也不同（详见下表）。对主界面槽位型功能（`value`/`key`/`id`/`display`/`sub`）而言，分别代表 `<mainId>` 与 `<槽位标识>`；对于其他功能则会调整为相应的主界面或子界面参数。
  * **`[|默认值]`**: (可选) 如果玩家没有对应的绑定，占位符将显示此处提供的默认值。

为了便于查阅，以下列出三类常见格式：

  * **主界面槽位型功能**：`%dgb_<功能>_<mainId>_<槽位标识>[|默认值]%`
  * **主界面绑定检测功能（has）**：`%dgb_has_<mainId>[|默认值]%`
  * **子界面条目检测功能（subhas）**：`%dgb_subhas_<subId>_<entryKey>[|默认值]%`

### 6.2. 功能类型详解

| 功能 (`function`) | 描述 |
| :--- | :--- |
| `value` | 获取绑定条目的 `value` 字段。这是最常用的占位符。 |
| `key` | 获取绑定条目的 `key` 字段。 |
| `id` | 功能与 `key` 完全相同。主要用于当槽位ID本身具有语义时，使占位符更易读，例如 `%dgb_id_character_panel_class_slot%`。 |
| `display` | 获取绑定条目在主界面上渲染后的**显示名称 (name)**。 |
| `sub` | 获取该绑定所属的子界面 `id`。 |
| `has` | 判断玩家在指定主界面上是否至少绑定了一个槽位。找到绑定返回 `1`，否则返回默认值（未指定默认值时返回 `0`；默认值仅支持 `0` 或 `1`，其它内容将按 `0` 处理）。 |
| `subhas` | 判断玩家是否绑定过指定子界面的某个条目。找到绑定返回 `1`，否则返回默认值（未指定默认值时返回 `0`；默认值仅支持 `0` 或 `1`，其它内容将按 `0` 处理）。 |

### 6.3. 使用示例

假设玩家在 `character_panel` 主界面的 `class_slot` (索引为11) 槽位上绑定了 `key` 为 `warrior`、`value` 为 `&c战士` 的条目。

  * **获取值 (value)**

      * `%dgb_value_character_panel_11%` -\> 返回 `&c战士`（最终输出为已润色文本）
      * `%dgb_value_character_panel_class_slot%` -\> 返回 `&c战士`（最终输出为已润色文本）
      * `%dgb_value_character_panel_11|平民%` -\> 如果玩家绑定了，返回 `&c战士`；如果没绑定，返回 `平民`（最终输出均为已润色文本）。
      说明：`value` 的最终返回值会在占位符解析后，统一经过 `ColorUtil.translateColors(...)` 进行颜色与样式润色。

  * **获取键 (key)**

      * `%dgb_key_character_panel_class_slot%` -\> 返回 `warrior`

  * **获取显示名称 (display)**

      * `%dgb_display_character_panel_class_slot%` -\> 返回 `&c[职业] &c战士` (根据 `overrides` 渲染后的名称)

  * **判断主界面是否存在任意绑定 (has)**

      * `%dgb_has_character_panel%` -\> 玩家在该主界面任意槽位存在绑定时返回 `1`，否则返回 `0`
      * `%dgb_has_character_panel|0%` -\> 如无绑定返回 `0`（默认值仅支持 0 或1）

  * **判断子界面条目是否被绑定 (subhas)**

      * `%dgb_subhas_profession_warrior%` -\> 若玩家曾在任意主界面槽位绑定过 `profession` 子界面的 `warrior` 条目，返回 `1`
      * `%dgb_subhas_profession_mage|0%` -\> 未绑定时返回 `0`（默认值仅支持 0 或1）

## 7. 数据存储 (Database)

### 7.1. 存储机制

  * 插件使用数据库来持久化存储所有玩家的绑定数据。
  * 支持 `SQLite`（默认，轻量级，无需额外配置）和 `MySQL`（推荐用于大型服务器或跨服场景）。
  * 数据库类型和连接信息在 `config.yml` 中配置。
  * 为了性能，插件会将所有绑定数据加载到内存中作为缓存。对数据的修改（绑定/清除）会先更新缓存，然后异步写入数据库，避免阻塞服务器主线程。

### 7.2. 表结构

插件会在数据库中创建一张名为 `gui_bindings` 的表。

**SQLite Schema:**

```sql
CREATE TABLE IF NOT EXISTS gui_bindings (
  player_uuid TEXT NOT NULL,
  main_id     TEXT NOT NULL,
  slot_index  INTEGER NOT NULL,
  sub_id      TEXT NOT NULL,
  entry_key   TEXT NOT NULL,
  entry_value TEXT NOT NULL,
  updated_at  INTEGER NOT NULL,
  PRIMARY KEY (player_uuid, main_id, slot_index)
);
```

**MySQL Schema:**

```sql
CREATE TABLE IF NOT EXISTS gui_bindings (
  player_uuid VARCHAR(36) NOT NULL COMMENT '玩家UUID',
  main_id     VARCHAR(255) NOT NULL COMMENT '主GUI标识',
  slot_index  INT NOT NULL COMMENT '槽位索引',
  sub_id      VARCHAR(255) NOT NULL COMMENT '子GUI标识',
  entry_key   VARCHAR(255) NOT NULL COMMENT '条目键名',
  entry_value TEXT NOT NULL COMMENT '条目值',
  updated_at  BIGINT NOT NULL COMMENT '更新时间',
  PRIMARY KEY (player_uuid, main_id, slot_index),
  KEY idx_gui_bindings_player (player_uuid),
  KEY idx_gui_bindings_player_main (player_uuid, main_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
```

| 字段 | 描述 |
| :--- | :--- |
| `player_uuid` | 玩家的UUID |
| `main_id` | 主界面的ID |
| `slot_index` | 主界面中的槽位索引 |
| `sub_id` | 子界面的ID |
| `entry_key` | 所选条目的key |
| `entry_value`| 所选条目的value |
| `updated_at` | 绑定的时间戳（毫秒） |

文件内的占位符（如 `{id}`, `{key}`, `{value}`）会在发送消息时被动态替换为实际内容。

## 9. 槽位类型系统 (Slot Type System)

### 9.1. 槽位类型概述

从最新版本开始，插件支持三种槽位类型，用于增强界面的交互性和美观度：

| 类型 | 适用范围 | 描述 | 是否可点击 |
| :--- | :--- | :--- | :--- |
| `DEFAULT` | Main/Sub | 默认类型，正常的绑定槽位或条目。 | ✅ |
| `DECORATION` | Main/Sub | 装饰槽，用于界面美化，仅显示物品。 | ❌ |
| `RETURN` | Sub only | 返回槽，点击后返回上一级主界面。 | ✅ |

### 9.2. 配置兼容性

**向后兼容：** 所有旧配置文件无需修改即可继续使用。未指定 `type` 字段时，默认使用 `DEFAULT` 类型。

**默认外观：**
- **装饰槽 (DECORATION)**：若未配置 `display` 或 `displayEmpty`，默认为灰色玻璃板 + 空白名称
- **返回槽 (RETURN)**：若未配置 `display`，默认为箭头 + "返回" 名称

### 9.3. 装饰槽 (DECORATION) 使用指南

**主界面装饰槽示例：**

```yaml
slots:
  - slot: 0
    type: DECORATION
    displayEmpty:
      material: GRAY_STAINED_GLASS_PANE
      name: ' '  # 空白名称，使其看起来更简洁
```

**子界面装饰槽示例：**

```yaml
entries:
  - slot: 8
    type: DECORATION
    key: "decoration_border"  # 仅用于内部标识
    value: ""
    display:
      material: BLACK_STAINED_GLASS_PANE
      name: ' '
```

**注意事项：**
- 装饰槽不需要配置 `sub` 字段（Main）
- 装饰槽不支持 `conditions` 和 `overrides`（Sub）
- 若 Main 槽位配置为 `DECORATION` 但仍指定了 `sub`，将输出警告并忽略 `sub` 配置

### 9.4. 返回槽 (RETURN) 使用指南

返回槽只能用于子界面，用于提供返回上一级主界面的功能。

**基础返回槽示例：**

```yaml
entries:
  - slot: 0
    type: RETURN
    key: "back_button"  # 仅用于内部标识
    value: ""
    display:
      material: ARROW
      name: '&f返回'
      lore:
        - '&7点击返回主界面'
```

**功能说明：**
- 点击返回槽后，会自动返回当前会话记录的主界面
- 如果会话信息丢失，会显示错误提示并关闭界面
- 返回成功后会触发 `messages.return-success` 消息

**错误处理：**
- 若会话数据丢失：显示 `messages.return-no-session` 并关闭界面
- 若主控制器未初始化：记录错误日志并关闭界面
