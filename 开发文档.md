# 开发总文档 — DrcomoGuiBinder（Main/Sub GUI 绑定模块）

---

## 1 项目总览

### 1.1 开发背景

本模块为 Drcomo 系列插件生态的 GUI 绑定子系统。目标是在 Minecraft 服务器中提供一套可配置、可持久化、玩家级独立的“主界面(Main)槽位 ←→ 子界面(Sub)条目”绑定机制，供其他玩法或插件通过占位符读取绑定值与展示效果。模块依托 DrcomoCoreLib 提供的 GUI、占位符、配置、异步与持久化工具实现。

### 1.2 问题定义与核心目标

* 问题：现有玩法需要为每个玩家配置若干槽位的“字符串变量”，且希望通过图形界面配置；配置需支持多 Main、多 Sub、Sub 根据被哪个 Main 使用选择不同展示。
* 核心目标：实现玩家级槽位绑定、Sub 控制绑定后展示、占位符解析、配置热重载、异步持久化与运维命令，保证安全性与性能。

### 1.3 功能简要概述

* 主界面(Main)定义空槽与关联 Sub；Sub 提供条目(entry)及其 `key/value` 与 `display`；玩家在 Main→Sub→选择后将 `entry` 绑定到该玩家的 Main.slot。
* Sub 决定绑定后 Main 槽的展示，支持 `overrides` 针对不同 Main 的显示覆盖。
* 提供占位符 `%dgb_<mainId>_<slot>%` / `%dgbk_...%` / `%dgbd_...%`。
* 提供命令、权限节点、事件钩子、SQLite 持久化、配置热重载、会话管理与日志等级控制。

### 1.4 项目目标与预期达成效果

* 功能交付：Main/Sub 双层绑定完成，配置规范稳定，持久化与占位符正确工作，命令与事件接口完备。
* 非功能：单服务器并发下每秒绑定操作能支持 >200 次短突发峰值；内存占用可控；无明显卡顿。
* 交付物：插件 jar、`Main/` 与 `Sub/` YAML 示例、配置样例、开发文档、用户手册、测试用例、数据库迁移脚本。

### 1.5 适用场景及目标用户群体

* 适用场景：游戏内槽位个性化、技能/装备变量选择、占位符值配置、跨插件数据提供。
* 目标用户：服务器管理员（配置、运维）、普通玩家（使用 GUI 绑定）、二次开发者（通过事件/占位符集成）。

## 2 模块功能结构

模块划分与职责如下，均以 `cn.drcomo.drcomoguibinder` 为根包名（示例）。

### 2.1 插件入口与总控

* 类：`cn.drcomo.drcomoguibinder.DrcomoGuiBinder`
* 功能：插件加载/卸载、各子模块初始化、命令注册、PAPI 注册。
* 输入：插件配置文件 `config.yml`、Main/Sub YAML。
* 输出：模块实例、命令与占位符可用。
* 约束：在 `onEnable()` 完成后其他模块方可使用。

### 2.2 配置服务（GuiConfigService）

* 类：`cn.drcomo.drcomoguibinder.config.GuiConfigService`
* 功能描述：加载 `Main/*.yml` 与 `Sub/*.yml`，校验结构并提供查询接口。
* 输入/输出：读取文件系统 YAML，输出 `MainGuiDef`、`SubGuiDef` 对象模型。
* 内部逻辑：文件监听（热重载），语法校验，ID 唯一性校验。
* 依赖：`cn.drcomo.corelib.config.YamlUtil`，日志 `DebugUtil`。
* 执行约束：热重载时暂时锁定配置快照，确保读/写一致性。

### 2.3 GUI 控制器

* 类：`cn.drcomo.drcomoguibinder.gui.MainGuiController`, `SubGuiController`（均继承 `cn.drcomo.corelib.gui.PaginatedGui`）
* 功能：渲染 Main/Sub 界面，处理点击事件，交互流程控制（打开/返回/超时）。
* 输入：玩家点击事件、会话上下文。
* 输出：打开/关闭界面、触发绑定事件。
* 内部逻辑：渲染时读取 `BindingService` 判断槽位是否已绑定；点击分发走 `GuiActionDispatcher`。
* 依赖：`GUISessionManager`, `GuiActionDispatcher`, `PlayerSessionManager`。
* 线程：渲染与事件均在主线程（Bukkit 线程）执行；持久化异步执行。

### 2.4 绑定服务（BindingService）

* 类：`cn.drcomo.drcomoguibinder.bind.BindingService`
* 功能：维护内存缓存的 `(player_uuid, mainId, slot) -> Binding` 映射，提供查询、写入、清理接口。
* 输入/输出：API 调用（bind/clear/get/list），输出 `Binding` 对象。
* 内部逻辑：写入时更新内存并异步提交到 `BindingRepository`；清理操作可同步或异步。
* 依赖：`AsyncTaskManager`（异步），`SQLiteDB`（持久化）。
* 约束：写入顺序保证（队列化），冲突写入按最后写入胜出（last-write-wins）。

### 2.5 持久化层（BindingRepository）

* 类：`cn.drcomo.drcomoguibinder.persist.BindingRepository`
* 功能：SQLite 操作封装，提供批量读写、迁移与备份接口。
* 输入：Binding 对象，查询条件。
* 输出：持久化结果/异常。
* 内部逻辑：使用事务、准备语句、批量提交；异常时回退并记录日志。
* 约束：必须在插件卸载前 flush 缓存；DB 文件权限需可写。

### 2.6 占位符扩展（PAPI）

* 类：`cn.drcomo.drcomoguibinder.papi.BinderPlaceholderExpansion`
* 功能：注册占位符解析器，响应 `%dgb_<mainId>_<slot>%` 等请求。
* 输入：占位符参数字符串、玩家对象。
* 输出：字符串值或默认值。
* 逻辑：解析请求、从 `BindingService` 取值、在渲染阶段可调用 `PlaceholderAPIUtil` 做二次解析。
* 依赖：`cn.drcomo.corelib.hook.placeholder.PlaceholderAPIUtil`。

### 2.7 命令与权限管理

* 类：`cn.drcomo.drcomoguibinder.command.CommandHandler`
* 功能：实现命令 `/DrcomoGuiBinder` 的子命令。
* 输入：命令参数，执行者权限。
* 输出：操作结果与反馈消息。
* 注册命令：`plugin.yml` 中声明并绑定到 `CommandHandler`。
* 权限节点（示例）：

    * `drcomo.guibinder.open`：打开任何 Main
    * `drcomo.guibinder.open.others`：为他人打开
    * `drcomo.guibinder.clear`：清除绑定（他人）
    * `drcomo.guibinder.reload`：重载配置
    * `drcomo.guibinder.save`：强制持久化

### 2.8 事件接口（对外）

* 事件类：`GuiBindEvent`, `GuiClearEvent`（包 `cn.drcomo.drcomoguibinder.event`）
* 功能：在绑定或清除时触发，允许其他插件监听并可取消或修改绑定结果。
* 触发条件：玩家在 Sub 点击后创建绑定，清理命令或 API 调用清除绑定时触发。
* 约束：事件处理在主线程，若监听器耗时需异步处理，否则可阻塞主流程。

### 2.9 辅助工具

* `ItemTemplateRenderer`：按配置渲染 ItemStack，支持 `{value}`、`{key}` 替换及 PAPI 二次解析。包路径：`cn.drcomo.drcomoguibinder.template.ItemTemplateRenderer`。
* `GuiSessionTimeoutManager`：处理 Sub 会话超时回退。

## 3 数据结构与状态管理

### 3.1 核心数据结构（对象模型）

* `MainGuiDef`

    * `id: String`, `title: String`, `size: int`, `slots: List<MainSlotDef>`
* `MainSlotDef`

    * `slot: int`, `sub: String`, `displayEmpty: ItemTemplate`
* `SubGuiDef`

    * `id: String`, `title: String`, `size: int`, `entries: List<EntryDef>`
* `EntryDef`

    * `slot: int`, `key: String`, `value: String`, `display: ItemTemplate`, `overrides: Map<mainId, OverrideDef>`
* `ItemTemplate`

    * `material`, `name`, `lore[], customModelData`, `flags[]`, `glow`, `nbtMap`
* `Binding`（运行时）

    * `playerUuid: UUID`, `mainId: String`, `slot: int`, `subId: String`, `entryKey: String`, `entryValue: String`, `updatedAt: long`

### 3.2 缓存体系

* `BindingService` 持有内存缓存 `ConcurrentHashMap<String(playerUuid+mainId+slot), Binding>`。
* 缓存写入：同步写内存；提交 DB 使用异步队列。
* 缓存加载：插件启动时从 DB 批量加载常用玩家（或按需加载）；建议按需加载策略（lazy load）。

### 3.3 会话机制（PlayerSessionManager）

* 作用：保存短期会话状态 `currentOpenMainId`, `currentOpenSlot`，用于 Main→Sub→回流时识别绑定目标。
* 生命周期：在玩家打开 Main 时创建，会话超时或玩家离线时销毁。超时配置可通过 `config.yml` 设置（示例 `sessionTimeoutMs`）。

### 3.4 状态生命周期管理

* 创建：玩家点击 Main.slot 后创建会话并打开 Sub。
* 绑定：玩家在 Sub 点击 entry 后写入 Binding（更新缓存并入异步写队列）。
* 失效：玩家下线、会话超时或管理员清除会话时移除会话。
* 回滚：DB 写入失败时在日志记录并重试，失败超限触发告警与管理员回滚策略（人工修复）。

## 4 配置与扩展机制

### 4.1 支持的配置格式与路径

* YAML（Main/Sub 文件）

    * `plugins/<plugin>/Main/<mainId>.yml`
    * `plugins/<plugin>/Sub/<subId>.yml`
* 全局配置 `plugins/<plugin>/config.yml`（YAML）
* 文本提示 `plugins/<plugin>/lang.yml`（YAML）

### 4.2 主要配置项与说明（完整字段）

**config.yml（完整示例）**

```yaml
resolveAtBind: false          # boolean: true=绑定时解析并固化 value；false=渲染时解析
sessionTimeoutMs: 90000       # int: Sub 会话超时毫秒
database: data.db             # string: sqlite 文件名
logLevel: INFO                # enum: DEBUG/INFO/NONE
async:
  writerThreads: 2            # 持久化写线程数
  writeBatchSize: 50
placeholders:
  prefix: dgb                 # 占位符前缀
```

### 4.3 Main/Sub YAML 字段（示例与解释）

* Main 文件：

    * `id, title, size, defaults`（可选），`slots: [{slot, sub, displayEmpty}]`
* Sub 文件：

    * `id, title, size, entries: [{slot, key, value, display, overrides:{mainId:{display}}}]`

### 4.4 扩展机制设计

* **事件注册**：公开 `GuiBindEvent`, `GuiClearEvent`，第三方插件可注册监听器以扩展行为。
* **命令扩展**：CommandHandler 采用子命令注册表，便于增加子命令。
* **占位符扩展点**：`BinderPlaceholderExpansion` 公开解析 hook，支持自定义解析策略。
* **热更新**：`GuiConfigService` 监听文件变更并触发热重载，重载时生成新快照并原子替换引用。
* **插件式注册**：对 `ItemTemplateRenderer` 提供接口 `TemplateRendererPlugin`，允许接入自定义字段或渲染器（例如自定义 NBT 注入）

## 5 功能实现说明（功能点与触发方式）

### 5.1 打开 Main 界面

* 使用方式：命令 `/DrcomoGuiBinder open <mainId>` 或游戏内触发器（管理员绑定）。
* 触发条件：玩家执行命令或触发绑定操作。
* 运行逻辑：

    1. 验证 `mainId` 是否存在。
    2. 构建 Inventory（调用 `PaginatedGui.createInventory`）。
    3. 对每个已配置 slot 渲染：若有绑定，调用 `renderBoundSlot()`；否则 `displayEmpty`。
    4. 打开界面（主线程）。
* 失败处理：若 `mainId` 不存在，返回用户提示 `open-no-exist`。

### 5.2 Main.slot 点击 → 打开 Sub

* 触发：玩家在 Main 界面点击特定槽。
* 运行逻辑：

    1. 通过 `MainSlotDef.sub` 找到 `subId`。
    2. 若 `subId` 不存在，返回提示并不打开 Sub。
    3. 在 `PlayerSessionManager` 注册当前目标 `(mainId, slot)`；打开 Sub 界面（\`SubGuiController.open(player, subId)）。
* 失败处理：Sub 不存在或权限不足返回相应提示。

### 5.3 Sub.entry 点击 → 绑定并返回 Main（核心流程）

* 触发：玩家在 Sub 点击 entry。
* 运行逻辑：

    1. 从 `PlayerSessionManager` 获取 `(mainId, slot)` 作为绑定目标；若未找到，提示并关闭 Sub。
    2. 构造 `Binding`：`{playerUuid, mainId, slot, subId, entryKey, entryValueProcessed}`。

        * `entryValueProcessed` 根据 `config.resolveAtBind` 选择：

            * 若 `true`：在绑定时使用 `PlaceholderAPIUtil` 解析 `value` 并存入数据库。
            * 若 `false`：存入原始 `value`，渲染时再解析。
    3. 发布 `GuiBindEvent`（可被取消或修改）。
    4. 将 binding 写入 `BindingService`（内存更新）并异步写入 DB（入写队列）。
    5. 关闭 Sub 并重绘 Main 对应槽（仅刷新该槽）。
* 失败处理：DB 写入队列满或异步写失败记录日志并重试；写失败达到阈值触发管理员报警。

### 5.4 清除绑定

* 触发方式：命令 `/DrcomoGuiBinder clear <mainId> <slot> [player]` 或通过 GUI 提供的“清除”按钮（若配置）。
* 运行逻辑：删除内存 binding 并异步删除 DB 记录；触发 `GuiClearEvent`。
* 权限：跨玩家清除需要 `drcomo.guibinder.clear` 权限。
* 失败处理：若 DB 删除失败，记录日志并在管理员界面显示失败项。

### 5.5 占位符解析（运行时）

* 请求格式：`%dgb_<mainId>_<slot>%`、`%dgbk_...%`、`%dgbd_...%`。
* 解析步骤：

    1. 解析 `mainId` 与 `slot`；若无对应 binding 返回空或默认。
    2. 取 `Binding` 的 `entryValue`（若 `resolveAtBind=false` 则在此阶段调用 `PlaceholderAPIUtil.parse(player, entryValue)` 返回最终字符串）。
    3. 返回字符串。
* 并发：占位符解析在主线程或调用线程执行，需保持高效、避免阻塞。

## 6 关键流程与交互逻辑（用户视角与系统视角）

### 6.1 用户视角流程（简洁步骤）

1. 玩家执行 `/DrcomoGuiBinder open weapons`。
2. 玩家点击槽 10（显示未绑定）。
3. Sub（sword\_vars）打开，玩家选择条目（伤害=12）。
4. 系统返回 Main，槽 10 变为 Sub 指定的展示（铁剑/名称“伤害=12”）。
5. 占位符 `%dgb_weapons_10%` 返回 `12`。

### 6.2 系统视角流程（详细有序）

#### 6.2.1 打开 Main

* Input: `open(mainId)`
* Steps:

    1. `GuiConfigService.getMain(mainId)` -> `mainDef`
    2. `Inventory inv = PaginatedGui.createInventory(mainDef.size, mainDef.title)`
    3. For each `slotDef` renderItem = `resolveSlotDisplay(playerUuid, mainDef, slotDef)`
    4. `GUISessionManager.openInventory(player, inv)`
* Errors: `mainDef == null` -> send `open-no-exist`

#### 6.2.2 点击 Main.slot

* Input: Click event with slot index
* Steps:

    1. `slotDef = mainDef.findSlot(index)`
    2. `subDef = GuiConfigService.getSub(slotDef.sub)`
    3. `PlayerSessionManager.set(playerUuid, mainId, slot)`
    4. `SubGuiController.open(player, subId)`
* Errors: `subDef == null` -> send `sub-no-exist`

#### 6.2.3 点击 Sub.entry -> Bind

* Input: Click entry index
* Steps:

    1. `session = PlayerSessionManager.get(playerUuid)` (contains target mainId, slot)
    2. `entry = subDef.findEntry(index)`
    3. `valueToStore = resolveAtBind ? PlaceholderAPI.parse(player, entry.value) : entry.value`
    4. `event = new GuiBindEvent(player, mainId, slot, subId, entry.key, valueToStore)` -> callEvent
    5. If event.cancel -> abort, notify player
    6. `BindingService.bind(playerUuid, mainId, slot, subId, entry.key, valueToStore)`
    7. `BindingRepository.asyncUpsert(binding)`
    8. `MainGuiController.updateSlot(player, mainId, slot)`
* Errors: DB queue full -> enqueue retry; repeated failure -> log and notify admin.

## 7 安全性与稳定性考量

### 7.1 并发与数据一致性

* 缓存使用 `ConcurrentHashMap`，写入通过单一异步写队列序列化到 DB，保证相同 `(player, mainId, slot)` 写序列化顺序。
* 读优先从缓存，缓存缺失时从 DB 同步加载并回填缓存（加载锁防止多个线程并发加载）。
* 写冲突策略：最后写入胜出（last-write-wins），若需复杂冲突策略可通过 `GuiBindEvent` 插件层拦截。

### 7.2 权限与越权防护

* 所有管理命令绑定明确权限节点。
* GUI 内的“为他人操作”仅在具备 `open.others` 权限时允许。
* 事件 `GuiBindEvent` 可被其他插件拦截以实施额外权限校验或经济消耗。

### 7.3 防刷与防误触设计

* 点击冷却：在 `config.yml` 设置 `clickCooldownMs`，同一玩家点击操作短时间内限流（默认 300ms）。
* 会话超时：如果 Sub 打开后超时未选择，自动关闭并清理 session。
* 点击重入保护：在处理绑定时将该玩家目标锁定，避免并发双点击导致双写。

### 7.4 异常恢复与资源释放

* DB 操作包装事务，遇异常回滚并重试（重试次数可配置）。
* 插件关闭：`onDisable()` 阶段执行 flush 缓存并等待写队列清空（超时后记录未写内容到 `data/failed-<timestamp>.json`）。
* 会话关闭：玩家下线自动清理会话并释放对象引用。

## 8 性能优化与可维护性设计

### 8.1 性能控制策略

* 延迟加载（lazy load）Binding：首次请求特定玩家的 binding 时加载对应玩家数据并保留一定时间（LRU 淘汰）。
* 异步批量写：BindingRepository 聚合批量写入（`writeBatchSize`），减少 DB I/O。
* 缓存热刷新：配置热重载只替换配置快照，不重建缓存。
* 占位符解析优化：在渲染阶段尽量避免重复解析相同字符串，可采用简单 per-render memo。

### 8.2 模块间低耦合设计

* 采用接口定义（例 `IBindingRepository`, `IGuiConfigService`），便于替换实现（如切换到 MySQL）。
* GUI 渲染、配置解析、持久化分别独立部署为服务类，清晰职责分离。

### 8.3 可升级与适配层

* 提供 `RepositoryFactory` 和 `TemplateRenderer` 插件接口以便接入不同存储或渲染策略。
* 版本兼容：Main/Sub YAML 加入 `schemaVersion` 字段，加载时进行兼容转换。

## 9 测试设计建议

### 9.1 推荐测试场景

* 主流程测试：打开 Main -> 点击槽 -> 打开 Sub -> 选择 entry -> 验证绑定写入缓存与 DB、界面刷新、占位符值。
* 边界值测试：slot 越界、无 sub 配置、entry 重名、多次快速点击并发。
* 异常路径测试：DB 写失败、配置热重载中点击、绑定事件被取消。
* 权限测试：普通玩家执行管理员命令应失败。
* 性能测试：并发模拟 1000 玩家随机绑定操作，测写队列延迟与内存占用。
* 回归测试：配置版本升级后保证旧绑定仍能正确解析或升级成功。

### 9.2 自动化测试策略

* 单元测试：Mock `GuiConfigService`, `BindingRepository`，测试 `BindingService` 逻辑与事件触发。

// MainGui render helper
ItemStack resolveSlotDisplay(UUID player, MainGuiDef main, MainSlotDef slotDef) {
  Binding b = bindingService.get(player, main.id, slotDef.slot);
  if (b == null) return renderTemplate(slotDef.displayEmpty, player);
  EntryDef entry = guiConfig.getSub(b.subId).findEntryByKey(b.entryKey);
  ItemTemplate tpl = entry.display;
  if (entry.overrides.containsKey(main.id)) tpl = merge(tpl, entry.overrides.get(main.id).display);
  tpl = applyReplacements(tpl, Map.of("key", b.entryKey, "value", b.entryValue));
  return ItemTemplateRenderer.render(tpl, player);
}

## 12 开发注意事项与交付清单

### 12.1 开发边界与协作

* 维护接口契约：`IBindingRepository`、`IGuiConfigService`、`ItemTemplateRenderer` 必须稳定以便集成测试。
* 异步边界：所有 DB I/O 必须异步，主线程仅做必要同步校验。
* 日志策略：敏感错误与异常在 ERROR 级别记录，操作类信息在 INFO，调试信息在 DEBUG。

### 12.2 部署与环境注意事项

* Java 版本：Java 17。
* 目标 MC 版本：1.18（兼容 Spigot/Paper）。
* 插件依赖：DrcomoCoreLib 必须在插件目录中并被 `depend` 声明。
* 文件权限：插件目录须允许 JVM 用户读写 SQLite 文件与配置文件。

### 12.3 监控与运维建议

* 定期备份 `data.db` 与 `Main/Sub` 配置。
* 在高并发活动期间监控写队列长度与 DB 延迟。
* 提供管理命令 `/DrcomoGuiBinder diagnostics` 输出写队列状态、缓存大小与慢日志。

### 12.4 交付产物清单

### 附：术语与约定（简表）

* Main：主界面定义文件（Main/<mainId>.yml）
* Sub：子界面定义文件（Sub/<subId>.yml）
* entry：Sub 的条目，包含 `key/value` 与 `display`
* Binding：实际绑定记录（player, mainId, slot -> subId, entryKey, entryValue）
* resolveAtBind：配置项，决定 value 在绑定时或渲染时解析占位符
